## Session Scope

This session covered the first implementation slices of the
`workflow-execution-persistence` epic:

```text
001 Database Bootstrap and Schema Versioning
002 Canonical JSON and SHA-256 Helpers
003 Core Execution Schema
004 Project and Workflow Persistence Methods
005 Workflow Run and Stage Plan Persistence Methods
006 Work Item and Queue Persistence Methods
007 Attempt Claim Transaction - running queued_at schema atom
007 Attempt Claim Transaction - claim contract atom
007 Attempt Claim Transaction - empty queue claim atom
007 Attempt Claim Transaction - successful claim transition atom
```

The implementation introduced a new `internal/persistence` package with a
controller-facing `OpenStore` boundary, a concrete `Store` type, and a
SQLite-backed database adapter that owns connection setup, foreign-key
enforcement, schema metadata creation, supported-version validation, and failure
on unsupported schema versions.

## Concepts Introduced

- A concrete `Store` type as the controller-facing persistence boundary.
- A database adapter boundary that contains SQLite-specific behavior.
- Store opening as the owner of schema validation and migration setup.
- Schema metadata as the first durable main-database contract.
- A distinction between controller lifecycle intent and raw SQL details.
- Deterministic JSON encoding through sorted object keys.
- Canonical JSON v1's integer-only numeric contract.
- Decimal values represented as schema-defined strings.
- SHA-256 hex validation as a small reusable contract.
- Core workflow-execution schema as DDL owned by the database adapter.
- Current placement tables separated from terminal outcome tables.
- Destructive replacement limited to the known metadata-only development schema.
- Store methods as task-oriented lifecycle/data operations rather than raw SQL
  access from controller code.
- Immutable source identity records that reject conflicting rewrites for an
  existing generated ID.
- Workflow run records as the durable submission boundary under an existing
  project/workflow source identity.
- Stage plans as ordered, immutable rows that can be retried idempotently when
  the submitted plan is identical.
- Active-run listing as a query over child stage states rather than a separate
  mutable run-status column.
- Logical work-item records separated from their current queue/running/terminal
  placement.
- Queue visibility as durable database state that can be reconstructed after
  restart without an in-memory queue authority.
- Status counts as derived data over placement and terminal tables, not a
  separate summary table.
- Queue timestamps as durable eligibility evidence rather than mutable
  scheduling metadata.
- Claim preparation as a schema-level Review Atom before adding the actual
  queue-to-running transaction.
- Claim request validation as a separate Review Atom from the claim
  transaction.
- Empty-queue claim behavior as a separate Review Atom from the successful
  queue-to-running transition.
- Successful claim as the first transaction that moves durable work placement
  from queued to running.

## Review Questions

1. Why should controller startup eventually call `OpenStore` instead of calling
   individual migration functions?
2. Why does the SQLite adapter reject an existing schema with no
   `schema_version` table?
3. What invariant does `PRAGMA foreign_keys = ON` protect once workflow
   execution tables are added?
4. What is the difference between `Store` as a concrete type and a broad
   repository interface?
5. Why does this slice avoid creating workflow, stage, work-item, and placement
   tables?
6. Why does canonical JSON sort object keys but preserve list order?
7. Why does canonical JSON v1 reject `1.5` and `1e3` as JSON numbers?
8. Why should decimal values be represented as schema-defined strings?
9. Why should fingerprint helpers live outside persistence and workflow
   packages?
10. Why does the schema use separate `queued_work`, `running_work`,
    `completed_work`, and `failed_work` tables instead of one mutable status
    column?
11. Why is destructive replacement limited to the metadata-only development
    schema?
12. What does `completed_work.skipped_parent_id` represent?
13. Why do project and workflow upserts reject conflicting values for an
    existing generated ID?
14. Why do delete-if-unused methods return a boolean instead of treating a
    missing row as an error?
15. Why is stage-plan insertion idempotent when every existing row matches?
16. Why should a run with no persisted stages still appear in the active-run
    list?
17. Why can active-run listing be derived from stage terminal states instead of
    a separate run status field?
18. Why does this slice persist `work_items` separately from `queued_work`?
19. Why is re-enqueueing an already queued item with the same `queued_at`
    idempotent?
20. Why should status counts be reconstructed from placement and terminal
    tables instead of stored in a mutable counter row?
21. Why should re-enqueueing an item with a different `queued_at` fail instead
    of silently updating or ignoring the timestamp?
22. Why must `running_work` preserve the original `queued_at` before the claim
    transaction is implemented?
23. Why should claim request validation be defined before the SQL transaction
    that uses it?
24. Why should an empty durable queue return a no-work result instead of an
    error?
25. Why must `ClaimNextWork` insert the attempt and running row before deleting
    the queued row and returning the payload?
