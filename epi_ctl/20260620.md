## Interactive Questions And Answers

### 1. Architecture Comprehension

Question: What responsibility did we add for fake HPCC, and why should that responsibility stay in the controller/runtime layer instead of the workflow definition?

Answer summary: The human was unsure. They guessed fake HPCC pretends to be a real HPCC to test things like SSH, and that it belongs in the controller because the controller requests compute resources and Slurm does that.

Evaluation: Partially correct at the architecture level: worker capacity belongs in controller/runtime, not in workflow intent. However, the answer incorrectly emphasized SSH, which was discussed as future direction but not implemented in this slice. It missed the concrete current pieces: fake `sbatch`, generated Slurm worker script, command-backed `hpcc` target, and fake run helper.

Score: 2/5

### 2. Data Flow Tracing

Question: Trace the fake-HPCC demo run from `scripts/fake-hpcc/run-demo` until a worker completes one `write_demo_output` item. Name the main process boundaries.

Answer summary: Client submits workflow to controller API. Controller compiles work items. Controller detects work but no workers. Controller calls HPCC module to create a worker. Worker gets work and writes output.

Evaluation: This is a coherent high-level trace and preserves the controller-worker pull model. It misses key boundaries added during the session: Bash helper starts controller binary, fake workflow variables select `hpcc`, controller prepares `.run/fake-hpcc/worker.slurm`, command-backed starter invokes `bash -lc ... fake sbatch`, fake `sbatch` runs the script, and the worker is launched through `go run ./cmd/worker`. The answer says "HPCC module," which is ahead of the implementation; current code is still command-backed and fake.

Score: 3/5

### 3. Failure Prediction

Question: Predict what breaks if the controller is started from Windows PowerShell but the fake worker script runs under Git Bash/Linux and uses `http://localhost:8080`.

Answer summary: `localhost` on the Linux subsystem refers to itself, not the Windows parent.

Evaluation: Correct core prediction. This was one of the actual failures seen during verification. The answer could be stronger by saying the worker fails to fetch `/work/next`, leaving the controller with pending work until the demo client times out.

Score: 4/5

### 4. Invariant Recognition

Question: What must remain true before the controller calls fake `sbatch` for the `hpcc` target, so the worker can actually start?

Answer summary: The human was unsure and guessed WSL must be running.

Evaluation: Weak. Important invariants include: the generated worker script must exist at `.run/fake-hpcc/worker.slurm`; fake `sbatch` must be callable through the configured command; controller and worker must share a reachable controller URL; worker config must point at that URL; the command-backed target must be `hpcc`; and runtime directories must be valid. "WSL must be running" is at most environmental context and not the controller/runtime invariant.

Score: 1/5

### 5. Reconstruction Ability

Question: Without exact syntax, sketch what `scripts/fake-hpcc/sbatch` does when called with `.run/fake-hpcc/worker.slurm`.

Answer summary: The human was unsure and guessed `sbatch` does not do anything except return true, while WSL acts as the pretend compute node.

Evaluation: Incorrect. The fake `sbatch` validates a script path, creates `.run/fake-slurm`, increments a fake job counter, logs submission metadata, prints `Submitted batch job <id>`, and runs the script either in foreground test mode or background mode. This is a significant reconstruction gap.

Score: 1/5

   ```text

   ```

3. Have the human manually edit one small test next session.

   Best target: a test for fake `sbatch` behavior or `WriteFakeHPCCWorkerScript`. This would force reconstruction of the concrete mechanics.

4. Before implementing SSH, require the human to answer:

   - What code changes when fake `sbatch` moves from local execution to SSH execution?
   - What stays unchanged?
   - Which layer owns upload/write/run?

5. Keep EC-3, but reduce slice speed.

   The small-slice process preserved commit boundaries, but recall still degraded. Add a one-paragraph human explanation checkpoint every 2-3 slices.

6. Do not treat passing fake-HPCC tests as ownership.

   Current score is below 10. The subsystem works, but retained mental compression is weak.
