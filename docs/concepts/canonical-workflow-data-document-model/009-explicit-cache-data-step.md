## Objective

## Data State Transition

```text
canonical cache_data step
    + one materialize alias
    + one resolved asset instance per fan-out item
    -> visible cache_data work item
    -> existing worker cache_data payload
    -> existing provider constraints
```


## Implementation Requirements

- Require `work.type: cache_data` for materialization work.
- Require `step.data.materialize` and reject compute-only parameters on the cache operator.
- Compile one visible work item per fan-out asset instance.
- Reuse current payload, worker operation, cache promotion, and provider resource constraints.
- Preserve normal sequential stage semantics; no hidden dependency generation.
- Reject duplicate explicit materializers for the same instance/domain within an unambiguous workflow plan, or document a deterministic conflict rule.
- Keep legacy `PlanCacheDataWorkItems` behavior only until OS-013 migration cleanup.

## Out of Scope

- Consumer hydration.
- Worker scope.
- Per-instance pipelining between stages.
- Removal of legacy behavior.

## Acceptance Criteria

- An explicit Yan-Roy header cache step compiles to visible cache_data work.
- Work-item IDs are stable from step/fan-out identity, not silently inserted into compute steps.
- Resource constraints attach to the explicit cache work.
- No compute work is generated by this operator.
- Existing worker cache_data tests remain green.
