- Controller worker execution framework: [`docs/concepts/controller-worker-execution-framework/README.md`](docs/concepts/controller-worker-execution-framework/README.md). `OS-001-one-by-one-worker-capacity-manager` is implemented: the controller evaluates durable queued/running/resource-blocked demand, records in-memory inflight worker-start reservations, starts at most one worker while unsaturated, confirms one reservation when a worker claims work, and reuses the configured execution environment or fallback local worker starter from controller launch configuration. `controller_config.worker_execution_pattern` also supports `null`, which keeps demand evaluation active while disabling worker process scheduling and inflight launch reservations. Worker-capacity evaluation now removes the temporary launch reservation and logs a skipped start when scheduling is enabled but `worker_target_environment` is not configured, avoiding phantom inflight capacity for manual-worker runs. Worker configs generated from a local `execution_environment` now carry absolute worker paths, can include `python_executable` and `max_asset_bytes`, and avoid recopying an unchanged local worker executable while a prior direct-process worker may still be running. `OS-002-landcore-workflow-admission-diagnostic-slice` is implemented: workflow admission failures now report the admission phase and underlying cause, and LandCore-shaped field-year-crop synthetic fan-out admission covers object accessors that bind scalar, path, and list parameters.
- Canonical JSON variable loading: [`docs/concepts/canonical-json-variable-loading/README.md`](docs/concepts/canonical-json-variable-loading/README.md). Placeholder concept for importing controller, project, workflow, and override JSON into implicit typed namespaces using canonical JSON value shapes.

- Sensitive variable propagation: [`docs/concepts/sensitive-variable-propagation/README.md`](docs/concepts/sensitive-variable-propagation/README.md). Operational slices `001-sensitive-metadata-and-safe-rendering`, `002-protected-reference-model`, `003-controller-envelope-and-persistence`, `004-worker-secret-resolver-and-redactor`, `005-trusted-go-workitem-sensitive-context`, `006-python-subprocess-secret-materialization`, `007-controlled-sink-redaction-tests`, and `008-credentialed-worker-fixture-smoke` are implemented. `009-concept-closure-and-doc-sync` is implemented. `internal/variable` variable declarations now accept `sensitive` and protected references, resolved values now carry sensitivity metadata plus safe redaction labels, and sensitivity propagates through references, aggregates, accessors, interpolation, and protected-reference declarations without exposing plaintext through default string or JSON rendering. Controller work-item persistence and `/work/next` assignment payloads now include a `goet/execution-envelope/v1` envelope that separates public parameter values from protected-reference metadata; sensitive plaintext work-item parameters and client-submitted sensitive plaintext run variables are rejected instead of being stored. The worker now has a `worker_env` protected-value resolver, redacted `SensitiveValue` formatting/JSON behavior, explicit plaintext access, and an attempt-local literal redactor for exact materialized secret strings. Trusted in-process Go worker dispatch now builds an `OperationContext` that separates public values from declared sensitive values, registers materialized plaintext with a redactor, and routes handler logging through a safe logger. Python subprocess execution can now materialize explicitly requested `worker_env` protected refs into env vars or restrictive attempt-local temp files, scrub captured stdout/stderr, reject `GOET_OUTPUT_JSON` containing exact materialized secrets before persistence, and remove temp secret files on success and failure. Sentinel tests now verify that the exact `goet-secret-sentinel-007-do-not-persist` value is absent from safe rendering, controller status JSON, worker failure errors, captured stdout/stderr, structured output persistence, and inspected persistence rows; artifact-content scanning remains documented as outside the phase-1 policy. A repeatable credentialed-worker fixture smoke runbook now verifies that `worker_env:GOET_FIXTURE_TOKEN` is resolved only in the worker process and that the raw `goet-fixture-secret-008-do-not-persist` value is absent from controller status, submission status, captured logs, worker output, controller logs, and SQLite persistence.
- Source-control resolution and cache: [`docs/concepts/source-control-resolution-and-cache/STATE.md`](docs/concepts/source-control-resolution-and-cache/STATE.md). Local source aliases such as `local:demo` now resolve from the configured controller repository cache under `local/sources/<alias>/files`; controller startup no longer hardcodes `local:demo` to the sibling `../go-etl-demo-project`. The controller `/workflow` endpoint accepts either source-reference project/workflow paths or inline project/workflow JSON documents and admits both through the controller repository cache before persistence. `goet submit` now embeds local `--project` and `--workflow` JSON files by default, includes local files declared by the workflow `source_manifest`, and sends source-reference paths instead when `--repo`/`--repository` is supplied.
- SSH refinement: [`docs/concepts/ssh-refinement/README.md`](docs/concepts/ssh-refinement/README.md). `OS-001-ssh-proxyjump-transport` is implemented: `SSHTransportConfig` accepts explicit `jump_hosts`, jump hosts can inherit the target identity or use hop-specific identity settings, each hop uses its own host-key policy, and `SSHTransport.Connect` builds the final target SSH client through the jump chain without invoking the external `ssh` command. Existing Exec, SFTP copy/list, filesystem helpers, runtime preparation, and Slurm submission continue to use the final target client, so gateway hosts are transit-only. `OS-002-reverse-controller-callback-tunnel` is implemented: `execution_environment.callback_tunnel` can bind a remote listener on the final SSH target or a selected jump host such as `jump_hosts[0]`, proxy callback traffic to the local controller, validate the worker-facing `controller_url`, keep the tunnel idempotently available across repeated worker-start preparation, and close the tunnel with the execution environment during controller cleanup. Preflight covers gateway-side HTTP reachability and, for `SlurmScheduler`, submits a tiny `sbatch --wait` compute job that uses `curl` to verify the worker-facing `/status` URL from a Slurm allocation. `OS-003-ssh-config-ergonomics-and-host-key-verification` is implemented: local `identity_file` and `known_hosts_file` values expand `~`, `$VAR`, and `${VAR}` before local file reads, `known_hosts` verification uses `golang.org/x/crypto/ssh/knownhosts`, jump hosts can inherit the target `known_hosts_file`, and `known_hosts` policy now requires an explicit or inherited file instead of falling back to insecure behavior.
- Secure network exposure of the GORC controller API: [`docs/concepts/secure-network-exposure-of-gorc-controller-api/README.md`](docs/concepts/secure-network-exposure-of-gorc-controller-api/README.md). `OS-001-controller-api-auth-contract` is implemented. `internal/controllerauth` now contains the first authentication contract model: it decodes a resolved `controller_config.authentication` object into disabled or bearer mode, validates phase-1 roles (`client`, `worker`, `admin`), rejects unknown authentication and credential fields, rejects duplicate credential IDs, and enforces exactly one token source declaration per credential. It also contains the pure controller route policy for `GET /healthz`, workflow submission, source bundles, submission status/logs, raw work submission, worker claim/complete/fail, controller status, observation logs, and shutdown. Credential material loading now reads `token_env` and `token_file` sources once, trims one trailing line ending, rejects empty and duplicate token material, rejects token files with group/other permissions when permission bits are available, stores only SHA-256 digests for matching, and avoids token contents in returned errors. Checked-in controller defaults now include disabled `controller_config.authentication` and `controller_config.controller_insecure_external_http_allowed = false`. Controller startup now resolves auth before opening the database, loads credential material once, rejects disabled auth unless both listen host and advertised `controller_url` are loopback, rejects external plain HTTP unless the explicit override is true, and wires the resolved auth mode, credential store, and route policy into controller middleware. `OS-002-controller-route-authorization-middleware` is implemented: bearer-mode requests pass through one route authorization boundary, `/healthz` remains public, unknown paths return `404`, protected routes reject missing or invalid credentials with `401`, wrong roles receive `403`, valid callers using unsupported methods receive `405`, disabled-auth development mode bypasses all route checks after OS-001 loopback startup validation, and authenticated principals are attached to request context for handlers that need them later. `OS-003-shared-controller-http-client` is implemented: `internal/controllerhttp` provides a reusable controller HTTP client with deterministic loopback URL validation, same-origin redirect policy, safe path construction, bearer token injection through a redacting `SensitiveToken`, bounded sanitized status errors that close unexpected-status bodies, and no dependency on CLI, worker, SSH, Slurm, or workflow semantics. `OS-004-cli-controller-credential-loading` is implemented in the current working tree: the CLI can load controller tokens from `--controller-token-file`, `GOET_CONTROLLER_TOKEN_FILE`, or `GOET_CONTROLLER_TOKEN`, trims one trailing line ending, rejects empty or oversized tokens, rejects broadly readable token files on Unix-like systems, migrates CLI controller requests through `internal/controllerhttp`, uses public `/healthz` for reachability, and avoids local controller auto-start for non-loopback controller URLs.
