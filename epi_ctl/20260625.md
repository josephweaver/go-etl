## Concepts Introduced

- Docker can remain the local outer runtime while SingularityCE becomes the inner HPCC-like runtime.
- The current local Slurm image is Rocky Linux 9, so it cannot install the Ubuntu Jammy `.deb` directly.
- The local test image replaces the inherited Apptainer package with SingularityCE because the packages conflict.
- A Docker worker image can be the build/development source for the later SingularityCE worker runtime.
- The current worker has no `--help` mode, so the smoke test uses a missing config path as a narrow executable-start check.
- `LocalTransport` satisfies the same `Transport` interface as the Docker transport, but copies and executes on the local machine.
- `DirectProcessScheduler` satisfies the same `Scheduler` interface as Slurm, but starts the worker executable directly.
- `SingularityWorkerRuntime` is not a scheduler; it rewrites the worker process command while leaving capacity acquisition to the scheduler.
- Direct-process jobs must outlive the HTTP request context after successful submission, matching the Slurm job boundary.
- Demo ledgers can cause skip behavior; reset the local demo ledger when the runtime directory is intentionally cleared.
- Runtime variables should avoid backend-specific names when the concept is shared by local, Slurm, and Singularity paths.
- Layer-specific settings belong in the layer's `worker_config` object; `worker_env` should stay reserved for system environment variables.
- File names should follow current ownership boundaries; Docker/Slurm-specific names should not hold generic worker launch resolution.
- The current controller worker-start path is `ExecutionEnvironment`; older `WorkerStarter` adapters should not stay once they are no longer called by production code.
- Required and optional typed resolver methods keep missing-variable behavior separate from wrong-type errors.
