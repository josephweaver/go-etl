## Interactive Questions And Answers

### 1. Architecture Comprehension

Question: In the current local path, what are the distinct responsibilities of the client, controller, and worker?

Answer summary: Client submits workflows, passes variables, starts controller if needed. Controller compiles workflows, maintains queue state, exposes workflow/status/shutdown/work APIs. Worker gets work, executes, reports success/failure, repeats.

Evaluation: Mostly correct. One precision issue: the current controller does not maintain a completed-work collection; it removes completed work from `assigned`. Completion history is not yet durable. Otherwise the component boundary is well understood.

Score: 4/5

### 2. Data Flow Tracing

Question: Trace `demo-workflow.json` from `go run ./cmd/demo-client` through to `cdl-demo-2024.txt` appearing under `cmd/worker/.run/data`.

Answer summary: Demo client deserializes workflow JSON, submits it to controller with override variables. Controller compiles workflow into pending work items. Worker grabs work items and creates `cdl-demo-2024.txt`. Answer also stated controller variable resolution resolves the path for the output file.

Evaluation: The high-level chain is correct. The path detail is partially wrong: the controller compiles `OutputFilename`, not the final worker-local `DataDir` path. The worker resolves the final path by joining its config `DataDir` with the work item's output filename. Good causal chain, but some boundary leakage remains between controller compilation and worker-local storage.

Score: 3/5

### 3. Failure Prediction

Question: If the worker starts from the repo root but its config paths are still relative to `cmd/worker`, what fails and why?

Answer summary: It may create files under a nested wrong path or fail because the path does not exist.

Evaluation: Correct prediction. The actual observed failure was directory validation failing for `.run/logs` when interpreted relative to the repo root. The answer captured both likely classes: wrong output location or missing path failure.

Score: 4/5

### 4. Invariant Recognition

Question: What must remain true for a future “skip existing output” optimization to be safe under the idempotency rule?

Answer summary: Same code, successful prior step execution, same input, and recorded hash/fingerprint.

Evaluation: Correct core invariants. Missing some precision around variable snapshots, resolved parameters, workflow/work-item identity, output fingerprint/manifest, and backend/runtime configuration. Still, the answer captured the most important correctness condition: existing filename is insufficient; skip requires successful matching execution under same code and inputs.

Score: 4/5

### 5. Reconstruction Ability

Question: From memory, sketch the controller’s worker-start decision after workflow submission: what inputs does it use, and why doesn’t it always start all possible workers immediately?

Answer summary: Controller starts `workers_per_start` at a time after minimum interval, constrained by min/max worker counts and pending work. It ramps organically because work-item duration is unknown.

Evaluation: Good reconstruction of the broad policy. Some naming drift (`workers_per_start` instead of current `worker_count_per_start`) is harmless. Missing one implemented invariant: organic scale-up waits for the previous worker to claim work by observing assigned count growth. Also the current code has a minimum-count path that can start immediately up to the floor. Overall operationally sound.

Score: 4/5

## Observed Weaknesses

- Completion state was described as maintained by the controller, but the current controller does not persist or retain completed work records.
- Output path responsibility was slightly blurred: controller compiles output filenames; workers combine those filenames with worker-local config paths.
- Worker-start reconstruction omitted the implemented confirmation rule that assigned work must increase before organic scale-up proceeds.
- The variable-based identity/fingerprint model is understood directionally, but not yet operationally grounded in tables, resolver lifecycle, or concrete generated values.

## Drift Indicators

- The codebase changed substantially in one feature branch: 23 files and +2531 net lines.
- Several new concepts were introduced together: workflow submission, client startup, controller shutdown, local worker startup, scaler state, idempotent output overwrite, and variable-based identity/fingerprint design.
- Current docs are ahead of implementation for SQLite, verified skip, generated runtime variables, and durable attempt metadata.
- There is mild terminology drift between discussion names and code names, such as `workers_per_start` versus `worker_count_per_start`.

## Recommendations

- Slow the next implementation step and keep it design-first: define the SQLite variable snapshot ledger before writing persistence code.
- Create a compact schema sketch that explicitly maps each table column back to a typed variable namespace/name/type.
- Add a `WHY_STATE.md` or a focused section in `PROJECT_STATE.md` explaining why IDs/fingerprints are variables rather than database-owned metadata.
- Before implementing skip logic, write prediction tests or table-driven examples for unsafe skip cases.
- Keep the next coding slice small: one interface or one table model, not full persistence plus controller integration.

## Audit Conclusion

The human can reason about the system at an operational level and can predict meaningful failure modes. Epistemic control is not lost, but the branch grew quickly and includes enough AI-authored structure that the next slice should intentionally reduce abstraction drift. The strongest next move is to freeze the identity/fingerprint semantics in a small SQLite variable-ledger design before adding database code.
