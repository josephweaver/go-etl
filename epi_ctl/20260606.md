## Concepts Introduced

### SQLite Through `database/sql`

Go code now uses the standard `database/sql` package with the pure-Go `modernc.org/sqlite` driver. The controller opens the SQLite DB, initializes schema, and closes the DB handle on shutdown.

### Transactions

`ledger.InsertAttempt` writes an attempt row and all attempt-variable rows in one transaction. This matters because a partial attempt snapshot would be worse than no snapshot.

### Controller-Owned Persistence

The database belongs to the controller process. Clients and workers still communicate only through HTTP. This preserves the intended runtime boundary.

### Variable-Native Config

Controller config is a JSON transport for variables. The loader normalizes all entries into `controller_config` so a controller config file cannot accidentally inject workflow, worker, or runtime scope.

### Completion Metadata

`WorkCompletion` can now carry attempt IDs, workflow/step IDs, fingerprints, code version, and timestamps. Those fields are optional so old `{"id": "..."}` completions remain valid.

### Placeholder Runtime Values

The current worker metadata is deterministic demo metadata, not final runtime identity. The next feature should replace those placeholders with generated runtime variables from the controller/worker lifecycle.

## Integration Surprises

- SQLite failed to open `.run/controller/ledger.sqlite` until `OpenSQLite` created missing parent directories.
- The local demo client readiness window was too short for cold `go run` startup, so readiness checks were increased from 3 to 10.
- The ledger path is relative to the controller process working directory. In the current demo, that means `.run/controller/ledger.sqlite` under the repo root.

## Understanding Questions

1. Why should only the controller open and write the SQLite database?
2. Why is the ledger an attempt snapshot ledger rather than a durable queue?
3. What makes `attempt_variables` more important than convenience columns like `work_item_id`?
4. Why does `/work/complete` still accept an `id`-only payload?
5. What is unsafe about using deterministic demo fingerprints for real skip decisions?

## Current Risks

- The completion metadata is still synthetic demo data.
- The ledger records only two runtime variables per attempt.
- Workflow/step/work-item runtime IDs and fingerprints are not generated by the resolver lifecycle yet.
- The in-memory queue is still not durable.
- Skip logic does not exist yet.

## Recommended Next Feature

Replace deterministic demo attempt metadata with generated runtime variables.

The next feature should probably start by defining where `runtime.workflow_instance_id`, `runtime.step_instance_id`, `runtime.work_item_id`, `runtime.attempt_id`, and fingerprints are generated and how they are passed into `WorkCompletion`.
