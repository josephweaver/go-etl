## Concepts Introduced

- Optional capability interface through type assertion:

  ```go
  type Preparer interface {
      Prepare(ctx context.Context) error
  }
  ```

- Execution environment decomposition:
  - transport reaches an execution platform
  - shell dialect renders commands/scripts safely
  - scheduler requests compute resources asynchronously
  - runtime describes the worker process layout after resources exist
- Difference between worker launch runtime and future internal work-item runtimes.
- Config-first construction of controller execution environments.
- Dockerized Slurm as local fake HPCC transport/scheduler fixture, not the final HPCC abstraction.

Answer summary:

Score: 4/5

### 2. Data Flow Tracing

Question:

> When a workflow submission has pending work and the controller decides to start capacity, trace the path from `submitWorkflowHandler` to the Slurm job submission.

Answer summary:

> Controller has `c.env` from config. `submitWorkflowHandler` calls prepare on execution environment. Runtime/dialect generate a Slurm script pointing to worker. DockerTransport copies worker and Slurm script, then executes Slurm script to start worker.

Evaluation:

Good high-level trace, but some causal details are compressed or slightly off:

- `submitWorkflowHandler` compiles workflow items, plans start count, then calls `startConfiguredWorkers`.
- `c.env.Prepare` prepares transport/scheduler/runtime before job submit.
- `WorkerRuntime.Prepare` creates directories, writes worker config, and optionally copies worker artifact when configured.
- The Slurm script is generated by `SlurmScheduler.Submit`, not by `WorkerRuntime.Prepare`.
- The Slurm script is copied by the scheduler through the configured transport, then `sbatch` is executed through the transport.

The mental model is operational but still blends runtime prep and scheduler submit responsibilities.

Score: 3/5

### 3. Failure Prediction

Question:

> If the Dockerized Slurm `slurmctld` container is not running, where should the failure surface now, and why?

Answer summary:

> In the DockerTransport execute command, because after prepare we assume `slurmctld` is running.

Evaluation:

Correct. More specifically, failure should surface first during `DockerContainerTransport.Prepare`, which calls `docker exec <container> true`, or later during runtime prep / scheduler submit if preparation were bypassed. This answer predicts the correct layer and failure mode.

Score: 4/5

### 4. Invariant Recognition

Question:

> What must remain true for `WorkerRuntime.Prepare` to safely prepare a worker job without hardcoding Docker-specific behavior?

Answer summary:

> `WorkerRuntime.Prepare` creates directories. If Docker containers are not running or wrong, it faults because exec is required. If containers do not have BashDialect it might fail.

Evaluation:

Partially correct but incomplete. The stronger invariants are:

- `WorkerRuntime` must use only the `Transport` interface for copy/exec, not Docker-specific commands.
- The transport must already target the correct execution platform.
- The dialect must localize paths and quote/render commands for the target shell.
- Shared paths must be visible to both scheduler/control container and worker jobs.
- Worker config paths used by Slurm script and runtime prep must match.
- Optional artifact upload must place an executable worker at the same path used by the Slurm script.

The answer focused on operational failure cases, not the abstraction invariant that keeps runtime transport-agnostic.

Score: 2/5

### 5. Reconstruction Ability

Question:

> Without naming exact line numbers, sketch the minimum config JSON fields needed for the controller to build the current execution environment.

Answer summary:

> Variables are not required per se. Need execution environment with transport, scheduler, runtime. Dialect is defaulted.

Evaluation:

This is materially incorrect on two points:

- Current controller config validation still requires `variables`.
- Dialect is not defaulted in `ExecutionEnvironmentConfig`; it must be present with type `bash`.

The minimum execution-environment config currently needs:

```json
{
  "variables": [...],
  "execution_environment": {
    "name": "...",
    "transports": [
      {
        "type": "docker",
        "settings": {
          "container": "slurmctld"
        }
      }
    ],
    "dialect": { "type": "bash" },
    "scheduler": { "type": "slurm" },
    "runtime": {
      "type": "worker",
      "settings": {
        "root": "/data/goetl"
      }
    }
  }
}
```

`controller_url` is required only if worker config should be written during runtime preparation.

Score: 2/5
