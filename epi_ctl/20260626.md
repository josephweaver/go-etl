## Questions Asked

1. What responsibility belongs to `SSHTransport`, and what responsibility should remain outside it in the controller/client setup flow?
2. Trace the intended flow from a user answering the SSH setup questionnaire to the controller eventually using SSH to prepare and run work on Fake HPCC.
3. Predict what would break first if a user generated an SSH key successfully, but the Fake HPCC container did not install that public key into `authorized_keys`.
4. What security/correctness invariant are we trying to preserve by requiring an explicit host key pin or trust decision instead of silently accepting any SSH host?
5. If `fake-hpcc-ssh-config.json` loads but the controller cannot connect over SSH, which files or components would you inspect first, second, and third?
6. How would you add the setup questionnaire to `demo-client` without putting controller transport logic into the client UI layer?
7. From memory, what are the main artifacts added or changed across features 012-014, and what does each one prove?

## Answer Evaluation

The answers showed a coherent high-level model:

- `SSHTransport` was correctly described as the controller-side transport for file/directory operations, permission changes, and remote command execution.
- The controller startup path was correctly understood as consuming an execution environment config that gives it enough information to prepare and launch workers through SSH.
- Authentication failure was correctly predicted as the first failure if the generated public key is not present in the target account's `authorized_keys`.
- The debugging order started with config, key material, and target-side authorization, which is the right operational radius before inspecting transport internals.
- The artifact reconstruction for 012-014 was concise and mostly correct.

Important gap:

- The answer described the questionnaire as if it already creates the target `authorized_keys` entry and a local `known_hosts` entry. The implemented feature 014 currently generates local key material and writes a generated controller config with host public key material, but it does not yet install the public key on the remote host or write a real OpenSSH `known_hosts` file. That mismatch is an epistemic drift indicator because it affects the actual operational readiness of the setup path.

- The target workflow includes automatic SSH key setup, public-key installation, and host trust capture, but the current implementation only covers the local setup engine and generated config.
- `PROJECT_STATE.md` and `TARGET_STATE.md` still describe the broader runtime direction accurately, but `PROJECT_STATE.md` has not yet been refreshed to describe the new SSH transport and client setup artifacts from this session.
- The distinction between a controller config's pinned host key and a real local `known_hosts` entry needs to be made explicit in future docs or implementation.
- The demo-client integration question remains open and should be the next concrete slice if feature 014 continues.

## Recommendations

- Next implementation slice should wire the setup engine through `cmd/demo-client` behind an explicit setup flag or subcommand.
- Before adding remote mutation, write down whether key installation is allowed to use password auth, an existing admin key, Docker exec for Fake HPCC only, or manual copy instructions.
- Add documentation that separates three artifacts: private key, target `authorized_keys`, and host identity pinning.
- Keep the next slice small: demo-client invocation and generated config path only. Do not combine it with remote authorized-key installation.
- Update `PROJECT_STATE.md` after the demo-client wiring so the current implementation state matches the new client setup path.
