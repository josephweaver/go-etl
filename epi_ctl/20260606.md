## Concepts Introduced

### SQLite Through `database/sql`

Go code now uses the standard `database/sql` package with the pure-Go `modernc.org/sqlite` driver. The controller opens the SQLite DB, initializes schema, and closes the DB handle on shutdown.

### Transactions

`ledger.InsertAttempt` writes an attempt row and all attempt-variable rows in one transaction. This matters because a partial attempt snapshot would be worse than no snapshot.

### Controller-Owned Persistence

The database belongs to the controller process. Clients and workers still communicate only through HTTP. This preserves the intended runtime boundary.

### Variable-Native Config

Controller config is a JSON transport for variables. The loader normalizes all entries into `controller_config` so a controller config file cannot accidentally inject workflow, worker, or runtime scope.

### Completion Metadata

`WorkCompletion` can now carry attempt IDs, workflow/step IDs, fingerprints, code version, and timestamps. Those fields are optional so old `{"id": "..."}` completions remain valid.

### Placeholder Runtime Values

The current worker metadata is deterministic demo metadata, not final runtime identity. The next feature should replace those placeholders with generated runtime variables from the controller/worker lifecycle.

### Questions And Answers

#### 1. Architecture Comprehension

Question: Why must SQLite access belong to the controller rather than the worker or client?

Answer summary: The controller holds variables and acts as the system source of truth. The DB is a persisted extension of that responsibility. The controller decides whether new work items are needed. Workers get fully resolved work items and do not need direct DB access.

Evaluation: Strong architectural answer. Correctly identifies controller ownership, variables as state authority, DB as controller persistence, and worker simplicity. Minor overstatement: the controller currently does not yet use the DB to decide whether new work items are needed; that is target behavior, not implemented behavior.

Score: 4/5

#### 2. Data Flow Tracing

Question: Trace one demo work item from `demo-workflow.json` through to a row in the `attempts` table.

Answer summary: Client deserializes workflow JSON and submits to controller. Controller compiles one step into two work items for 2024 and 2025. Worker accepts work, does work, calls `/work/complete`, and controller records completion in `attempts`.

Evaluation: Correct high-level flow, but includes two important implementation inaccuracies. First, generated pending work items are not persisted in the DB; only completed attempts are recorded. Second, assigned state is in the in-memory `assigned` map, not the DB. Good causal chain, but the persistence boundary is blurred.

Score: 3/5

#### 3. Failure Prediction

Question: If the worker sends only `{ "id": "work-item-001" }` to `/work/complete`, what happens to the queue and ledger?

Answer summary: Predicted that the work item would be removed from assigned and the attempts record updated as successful, but output variables/fingerprint data would be missing for downstream steps.

Evaluation: Half correct. The queue prediction is right: if the item is assigned, the controller removes it from `assigned`. The ledger prediction is wrong for current implementation: `id`-only completions are accepted for backward compatibility but do not create or update a ledger attempt. The answer correctly identifies that missing output/fingerprint data makes future downstream/skip decisions impossible, but it projected future ledger update behavior that does not exist.

Score: 2/5

#### 4. Invariant Recognition

Question: What must be true before a prior ledger attempt can safely justify skipping future work?

Answer summary: Prior steps/work items must be complete with valid outputs; the step/work-item compile hash must match; future file hashes/input file changes must force rerun.

Evaluation: Good correctness instincts. Correctly rejects filename-only skip and names prior completion, valid outputs, matching compiled identity/fingerprint, and input-file hashes. Missing some specific invariants: code version, resolved variable snapshot, backend/runtime-relevant config, output fingerprint or manifest, and prior successful status. Still a strong answer.

Score: 4/5

#### 5. Reconstruction Ability

Question: Sketch the current namespace precedence order at a high level.

Answer summary: Globals, client env/config, controller env/config, worker env/config, project, workflow, override, step, work item, runtime.

Evaluation: Mostly correct and captures the important shape: broad config/env first, then project/workflow/override, then lifecycle scopes, then runtime. Exact order is slightly off: current canonical order places all env scopes before all config scopes: `global_config`, `client_env`, `controller_env`, `worker_env`, `client_config`, `controller_config`, `worker_config`, `project_config`, `workflow`, `override`, `step`, `work_item`, `runtime`. The answer interleaves each actor's env/config pair. This is a moderate precision gap, not a conceptual failure.

Score: 4/5
